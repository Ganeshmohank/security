---

# Why this project

**Problem.** Python code ships to production with the same class of bugs
year after year: SQL injection, hardcoded credentials, unvalidated input,
unsafe dynamic execution. A single-pass code review misses most of them.

**Objective.** Build a command-line tool that scans Python source,
combines multiple analysis engines, and produces a CI-friendly report
that flags vulnerabilities before they are merged.

**Why it matters.** The CMPE-279 brief names `ast`, `bandit`, and
`radon` for a reason — each catches a different failure mode. A tool
that fuses them gives the reviewer one place to look instead of three.

---

# What we built

**secscan** — a Python package + CLI that runs **four independent
engines** over a codebase and cross-validates their findings.

Highlights:
- **7 custom AST rules** including lightweight taint tracking for SQL
  injection (`sql = f"..."; cur.execute(sql)`).
- **Curated bandit subset** (B201, B506, B602, B701).
- **Radon complexity engine** — flags hard-to-audit functions.
- **Optional LLM engine** via OpenRouter for context-aware review.
- **Cross-engine confirmation** — findings reported by ≥2 engines are
  tagged "confirmed" for the reviewer.
- **Three report formats** — text, JSON, and a zero-JS HTML dashboard
  with collapsible file sections and source-code snippets.

# Architecture

```
    +--------+     +-----------------+     +----------------+
    |  CLI   | --> |  core/scanner   | --> |  report/builder|
    +--------+     +-----------------+     +----------------+
                           |
         +--------+--------+--------+--------+
         |        |        |        |
    +---------+ +--------+ +--------+ +--------+
    |  rules  | | bandit | | radon  | |  llm   |
    | (AST)   | | (proc) | | (lib)  | | (API)  |
    +---------+ +--------+ +--------+ +--------+
                    \     |    |    /
                     v    v    v    v
                 core/finding (shared dataclass)
```

- **`core/finding`** — single shared `@dataclass` every engine emits.
- **`core/scanner`** — walks files, fans out to engines, dedupes.
- **Engines** — each owns one analysis source; fully swappable.
- **`report/builder`** — renders text / JSON / HTML from the same data.

# Implementation details

**Tech stack.** Python 3.11+, `ast` stdlib, `bandit`, `radon`,
`requests` (OpenRouter), `pytest`.

**Design decisions.**
- **Rule-registry pattern** — each check is a `Rule` subclass appended
  to a list. Adding a rule = one file edit.
- **Shared `Finding` dataclass** — lets teammates work in parallel
  without merge conflicts.
- **AST-based secret detection** — replaced raw-text regex to kill
  false positives inside f-string SQL queries.
- **Native `<details>` HTML report** — zero JavaScript, single file,
  safe to email.

**Challenges we solved.**
- Secret rule firing on `password = '{raw}'` inside f-strings → fixed
  by walking `Assign`/`AnnAssign` nodes instead of scanning source text.
- Cross-engine dedupe missing obvious duplicates → fixed by folding
  `detail` text into the dedupe key.
- Broken virtual environment from a prior user → rebuilt with `venv`.


  engine with grade thresholds, scanner orchestration, and all three
  report renderers.

# Conclusion

**What works.**
- One command produces a clean, reviewer-friendly report that fuses
  four analysis sources.
- `--fail-on high` makes the tool drop-in ready for a CI pipeline.
- Cross-engine confirmation concentrates reviewer attention on the
  findings least likely to be false positives.

**Impact.** A two-person team delivered a tool that materially extends
the state of the art for a typical class project: taint-lite analysis,
multi-engine fusion, and a polished HTML report — not just a wrapper
around `bandit`.

**Future work.**
- GitHub Actions workflow + SARIF output for PR annotations.
- Pre-commit hook.
- Cross-function taint tracking.
- Language support beyond Python (JS, Go).
